GOALS FOR MY PROJECT
1. Reduce LLM token costs by 70-90% using semantic similarity caching
2. Implement advanced token control: dynamic truncation with tiktoken
3. Use modern tools: FastAPI, Redis, Chroma, tiktoken, LangChain, Docker and deeply understand their roles

SEMANTIC CACHING:
The traditional caching works as follows:
• storing exact query → response.
    If user asks: "What is your return policy?" → cached.
    If user asks: "What's your return policy?" (different wording) → cache miss → LLM call → tokens burned.
    The Semantic caching works as follows:
• storing EMBEDDING of query → response.
    Both queries have similar embeddings → cache hit → instant response → zero tokens used.

SEMANTIC CACHING WORKFLOW:
1. Client POST /chat with { "query": "How do I return a defective product?", "context": "optional prior history" }
2. FastAPI receives, creates a unique cache key = (context + query)
3. Compute embedding of key using same embedder (all-MiniLM-L6-v2)
4. Search Chroma for similar embeddings (cosine similarity)
   - If similarity >= threshold (e.g., 0.92) → cache hit
     - Return stored response immediately (latency ~10ms)
     - Log cache hit, tokens saved = length_of_stored_response
   - Else → cache miss
     - Apply token control: check if prompt exceeds max_prompt_tokens. If yes, reject with HTTP 400 error; otherwise proceed.
   - Call LLM (async, with streaming optional)
     - Store in:
       a) Redis (exact key → response, TTL) for exact-match speed
       b) Chroma (embedding → response metadata) for semantic search
     - Return LLM response
5. Response includes metadata: cache_hit, tokens_used, tokens_saved, latency_ms

TOKEN CONTROL WORKFLOW:
• Use tiktoken to accurately count prompt tokens.
• If token_count > max_prompt_tokens, reject the request immediately with HTTP 400 and a clear error message:
   "Prompt too long (X tokens, limit Y). Please shorten your input."
• No truncation is performed. The user is responsible for shortening the prompt.
• This ensures data integrity and transparency – no silent alteration of the user's input.

ECONOMIC ADVANTAGES:
1. Cost reduction – Each cache hit saves $0.000X per call; at millions of queries, it's huge.
2. Latency reduction – 2 seconds → 10ms for repeated/similar questions.
3. Consistency – Same answer for similar questions (no model drift or slight hallucinations).
4. Scalability – Cache servers can handle 100x more QPS than LLM endpoints.

EXPLANATIONS OF THE VARIOUS FILES:

REQUIREMENTS.TXT
We will use 5 categories of tools:
1. Web API → Expose the cache as a service (REST endpoints) → FastAPI + Uvicorn
2. Caching → Store exact responses (Redis) and semantic embeddings (Chroma) → Redis, ChromaDB
3. LLM & Embeddings → Generate answers (Groq) and convert queries to vectors (local embedder) → LangChain Groq, sentence-transformers
4. Token Control → Count and truncate prompts to fit model limits → tiktoken
5. Configuration & Utilities → Manage settings, retries, logging → pydantic-settings, tenacity, loguru
Why these are modern and production-ready:
• FastAPI – async, auto-OpenAPI docs, Pydantic validation – used by Netflix, Uber.
• Redis – industry standard for low-latency caching (used by Twitter, GitHub).
• Chroma – lightweight vector DB, but can be swapped for Qdrant/Weaviate later.
• Groq – LPU inference (very fast), free tier, used by AI startups.
• tiktoken – OpenAI's tokenizer, exact token counting (no estimation).
• loguru – prettier, more informative logging than Python's built-in.
Additional notes on this file:
• uvicorn – ASGI server that runs FastAPI
• redis – In-memory key-value store. We'll use redis.asyncio for non-blocking operations. Stores exact cache: (query_hash → response).
• chromadb – Vector database. Stores embeddings of queries plus response metadata. We'll search for similar embeddings using cosine similarity
• langchain-groq – Official Groq integration for LangChain. Wraps Groq's API with the standard LangChain BaseLLM interface
• sentence-transformers – Library for sentence embeddings (from Hugging Face). We'll load all-MiniLM-L6-v2.
    This runs locally, no API needed, uses ~80MB RAM. It converts queries into 384-dimensional vectors for similarity search.
• tenacity – Retry library with decorators. We'll wrap the Groq API call with @retry to handle transient network errors
• tiktoken – OpenAI's fast tokenizer. Even though we're using Groq, we can use the cl100k_base encoding (GPT-4's tokenizer)
    as an approximation for any modern LLM. It counts tokens accurately, allowing us to truncate prompts to stay within Groq's context limit (e.g., 8192 tokens)
• loguru – Modern logging with structured output, colors, and easy file rotation. We'll replace the standard logging module.


CONFIG.PY
In any production system, configuration must be separate from code. This allows:
• Changing settings without modifying code (environment variables, .env file)
• Running the same code in development, staging, and production
• Keeping secrets (API keys) out of version control
1. About Redis (REmote DIctionary Server): we use Redis when a user asks exactly the same question (byte-for-byte same string),
     we want to return the cached response in ~10ms without going to Chroma or LLM.
    We use Redis in Docker: redis://localhost:6379 and we put this in config.py, instead of hardcoding, allowing us to switch
     between environments, for example Development and Production, seamlessly, without changing code.
    With redis_ttl_seconds, we decide how long cached responses live, otherwise we would endlessly grow our memory, but, without, we would
     lose the main purpose of caching for FAQ; TTL forces periodic LLM to refresh cache but keeping repetitive ones, at least temporarily.
     In our project we set it at one hour, however, being in config.py, we can seamlessly change it, in regard to production demand.
     Just to keep in mind, once the cache expires, next request will be a cache miss, forcing the LLM to give a fresh response.
     Possible Redis enhancements, may include "socket_timeout", "retry_on_timeout", "health_check_interval".
    With redis_max_connections, we establish a max in the connection pool size; if the limit is reached, the next requests will wait for a "socket" to open.
     It is written in config.py in order to then customize it, with the business needs in production.
     On a laptop 10 is fine; in a serious production environment, it can be easily scaled over 100.
2. About Chroma: Chroma is a vector database. In our semantic cache, we store a KEY, namely an embedding vector, and a VALUE, namely the original query
      text + the LLM response.
     When a new query arrives, Chroma computes its embedding through the LLM; asks Chroma to find "the most similar stored embedding" through
      similarity search (for example cosine similarity) and, if "similarity score >= similarity_threshold", we get a cache hit, hence we return the
      cached response. Other vector embedding, eventually more powerful, are Qdrant or Pinecone or Weaviate.
     With chroma_persist_directory, it says to Chroma where to store its data to then load it from the given directory; this is consistent
      through restarts. In production, a Docker volume or a mounted permanent disk can be alternatively used. These settings are more easily
      configurable through the config.py file. The defualt location is a "./chroma_cache" local folder in the project root.
     With similarity_threshold, we define the minimum requirement to achieve the cosine similarity hit, therefore the cache hit. Empirically,
      0.92 is the sweet spot for short queries (i.e. customer questions), balancing safety and hit rate. Must be a value between 0.0 and 1.0.
      Other values raise a validation error at startup.
     With embedding_model_name, it specifies the LLm to use for converting sentences/text into embeddings; the model runs locally and produces
      vector of fixed dimensions to feed for the similarity search. Good models to use, given existing benchmarks: all-MiniLM-L6-v2 (lightweight,
      ~80MB RAM, 384-dim vectors, with fast inference), all-mpnet-base-v2 (higher quality but bigger dimension 768-dim),
      BAAI/bge-small-e (good for retrieval).
     CONCERN: Unlike Redis, Chroma does NOT have a TTL (Time To Live), hence it can scale memory consuption scarily fast in production-like environments,
      deletion is managed by-hand by default (concern!). Can be "hybridized" into a Redis like setting, we explain possible strategies:
       2a. 1st strategy: Time-based Metadata + Periodic cleanup (consistent but objectove and non concept related)
         We store a timestamp, added entry in metadata, for each entry and run a background job (i.e. daily) to delete entries older than N days.
       2b. 2nd strategy: LRU (Last Recently Used), namely keep only last N entries (More complex, but more productyion solid)
         We store up to N entries and gradually delete older ones upon addition of new ones
       2c. 3rd strategy; Using more sophisticated tools
         More complex but complete tools can be used intead of Chroma, namely Qdrant (support TTL), Weaving (supports time-based sharding and cleanup),
          Pinecode (can set delete by metadata filter)
     We used the LRU strategy
     Nonetheless, the vector store can be simply changed later one without changin the logic (abstraction) of the project.
3. About Token control: once we hit a cache miss (exact and sematic cache check), the LLM is called and the input may contain the User query,
      additional context from the conversation and system instructions. If the total context exceed the mox context window (N tokens for each model),
      the LLM may either refuse the request or truncate silently and unexpectedly; this is awful and disingenuous in production. Moreover, sinding long
      prompts wastes tokens (cost) and increases latency (slowness). A common way to assess this problem are a "smarter" truncation
      (namely keeping only the first 60% and last 40% of the prompt), which still acts silently towards the User.There could be more stable and "ethic"
      ways for an honest and stable production-style solution:
       3a. 1st strategy: Alert and Reject (more critically exact for the User, useful where data loss in not acceptable)
         If token count > max_prompt_tokens, it does NOT call the LLM; but it reponds clearly stating to the User to shorten the prompt.
          The response will be like Return HTTP 400 with error: "Prompt too long (X tokens, limit Y)"
       3b. 2nd strategy: Smart sliding window (better for conversational systems where older context loses importance)
         The memory only keeps the most recent N tokens and drops older context, i.e. chat memory
     We used the Alert and Reject strategy.
     The Token control will:
       1. Counts tokens, measuring accurately prompt length through tiktoken
       2. If len(prompt_tokens) > max_prompt_tokens, it will raise a HTTPException with details and NOT call the LLm, otherwise, the LLM call goes on.
4. About Perfomance: it will handle error management, namely:
     • request_timeout_seconds: float = Field(default=30.0, ge=1.0) - sets the max time (in seconds, this case 30) for the HTTP to send a response
        from Groq after sending the request. If the time exceeds, the request is canceled and an exception is raised.
        Good and customizable for production-grade scenarios.
     • retry_attempts: int = Field(default=3, ge=0) - sets the max retry attempts for a LLm request before giving up, and, therefore, raising an exception.
     • retry_backoff_factor: float = Field(default=1.0, ge=0.1) - controls the delay between retries, using the formula
            delay = backoff_factor * (2 ^ (retry_number - 1))
        hence:
            Retry 1, wait one seond; Retry 2, wait 2 seconds; Retry 3, wait 4 seconds
5. About Model Config: this class is the configuration for Pydantic's Base Setting, it controls how environment variables and .env files are loaded,
      and how extra fields are handled. In particular, it says where to look for environment variables. By default, it reads from os.environ.
      With env_files, it also reads from a .env file.
     The extra part will say how we define extra fields, namely if we ignore, warn or raise an error.
     The encoding ensures the .env file is read withUFT-8 (critical for non-ASCII characters).
     This is standard for professional Python projects.
     This layout ensures that secret data can be injected directly from private production settings, where secrets are stored.
     It is provided a .env.example dummy file with example values and instructions.
     If no .env file is found, the script simply skips it, no error, and it reads from system environment variables.

CACHE/EXACT.PY • END FOCUS ON FASTAPI
This file defines the exact cache (Redis) implementation. This is the first cache layer,
  that will hit once the same question is repeated; in this case, the previous response is recycled, and, therefore,
  no LLM tokens (no cost) are used and the latency is critically reduced (~10ms).
  We recall that Redis looks in-memory, hence lookups tak microseconds (low latency); it supports TTL (Time-To-Live) rules,
  so old queries expire automatically and, lastly, can be shared across multiple service instances, always natively,
  which makes it good for scaling into production-like complex systems.
It reasonably uses SQL-like commands, standard for query operations, and to manipulate relational databases.
 The operations used in this project are
 • get(query, context) - GET to fetch if the key exists, returning a cached response or NONE;
 • set(query, context, response) - to store the response with TTL (from config);
 • delete(key) (optional) – to manually DELETE keys in case of cache invalidation;
 • stats() – this returns the number of keys, useful for monitoring.
For storing a Redis keys, we compute a SHA256 has made of (context, query), this allows to keep keys short and deterministic.
 REMARK: SHA256 HASH: short for Secure Hash Algorithm 256-bit, is a member of the SHA-2 family,
  and it produces a unique 256-bit (32-byte) hash for any given input.
To not-blocking async operations, which is critical for FastAPI, we use redis.asyncio.
To store the response, we use either JSON or plain text (plain text is enough in this project, but can be customized easily).
The TTL is set up from settings.redis_ttl_seconds, the default is 3600 seconds = 1 hour.
Lets us analyze the code:
 • def _make_key - used to create the keys to memorize for the queries; given they are made of strings (possibly long),
    we convert them to a fixed 64-char hex string (memory cost-efficient).
    We use a prefix "chat:*" in order to logically group the exact caches from other instances (for exmaplke semantic caches) or other keys.
    Moreover; if we add keys in the future, this setup will NOT make them collide with the exact caches ones,
    it will be needed to use another prefix in case (e.g. session:xyz789, key for user sessions, or,
    rate_limit:user:42, key for rate limiting for users). Lastly, this allows us to edit them or act with them,
    using them as a group: for example, we can delete them all at once, using the prefix; we can add other prefix levels,
    for example, chat:development:..., chat:quality:..., chat:production:...; it allows us to read them better,
    as the prefix will indicate whoever is auditing the code what are the keys about.
 • async def get - tool to retrieve with cache: it retrieves cached response for exact query+context and returns a
    response string if cache hit and None otherwise.
    We note that - logger.DEBUG - means that we log at DEBUG level, hence visible only when LOG_LEVEL = DEBUG.
 • async def set - tool that stores the response with exact caching, namely stores it with the 64-char hex string key,
    with its TTL, taken from setting (aka config.py).
    The response is stored as a plain string (enough for the project), but can be customized to be stored as a JSON output too.
 • async def delete - tool to manually delete an exact cache entry and returns True if the key existed, and, therefore,
    has been deleted.
    This is crucial on the managing of the project, allowing the dev to manually purge entries at will, with the key.
 • async def stats - returns the precise amount of exact cache stored; as the previous one, great for monitoring.
    We scan only for keys with our prefix "chat:*", for consistency.
    We note that this setup is consistent for a relatively small number of keys (~100k keys);
    for greater amounts it's better to use the logger at INFO level or a separate "manual" counter,
    this would be less clean as it would not naturally pair up with the TTL expiration setup.
    This choice with the stats part is good ONLY if we assume the cache rate is small enough,
    otherwise the run of the stats part could grow linearly, but yet, very fast, if there is a huge amount of users
    (mind TTL is NOT based on amounts, but on time!); so, it is good to keep this in mind, as a yellow flag, in case of serious scaling.
    Alternatives (more complex as non-redis native for TTL) are redis INFO (we need to subtract keys from other namespaces (non "chat:*" ones);
    create a counter, which has to keep into accounts the keys that have been deleted through TTL, as it does NOT trigger a decrement automatically.
    Still a limitation to keep in mind.
    REMARK: count=1000 - does NOT mean that it wiull count only up to 1000 keys! It is a HINT to redis on how many keys count for each "round":
      if there is 50k keys, it will iterate the command 50 times (with 1000 keys each lap), hence making 50 round trips back and forth to Redis.
     Still, Redis, may do more or less than the exact amount, it is not a rigid contraint.
     Note that setting the count too slow, means too many Redis round trips, hence very slow; too high count and Redis may have issues
     while building a huge reply, and, if unlucky, block, creating memory and performace spikes. Generlly, 1000 is a reasonable default.
 • def __init__(self) - and - async def close - to start or end our connection pool to the FastAPI to Redis.
REMARK: FastAPI: FastAPI is a modern Python web framework for building APIs and its key features are:
 • Async: supports async def endpoints, enabling high concurrency (and, therefore, reduced latency);
 • Pydantic integrations: request/response validation, serialization and documentation;
 • High perfomance: reduced egenral latency.
 • Horizontal scaling: allows multiple istances to run on it simultaneously;
 • Vertical Scaling (more power, CPU/RAM-use, per instance): it can handle thousands of concurrent connections and,
    with univorn as additional tool, can handle multiple co-workers (tools), where, every worker has a separate process.
 This whole project is aimed at attacking the principal bottlenecks present in tools existing: LLM calls
  (expensive as end-month "token bill") and latency optimization, through cache hits. In bigger production projects, Chroma
  (good for dev environments) can be easily and smoothly swapped for a distributed vector DB, depending on needs, budgets and project needs.
